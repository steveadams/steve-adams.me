# Structural Guardrails for LLM-Generated Code

Here's the recursive observation at the center of this series: the same structural patterns that make LLM harness behavior deterministic also make LLM-assisted development reliable. These aren't two separate problems with a convenient overlap. They're the same problem — the absence of structural constraints — manifesting at different levels.

**Runtime.** Your application uses an LLM. Its output is non-deterministic. The model returns structured responses that might be malformed, tool calls with invalid arguments, or hallucinated function names. Structural patterns constrain what your harness can do with that output — parsing it into typed data or rejecting it, never silently passing it through.

**Development.** An LLM helps you write code. Without a clear spec or existing conventions, it eventually turns into a really shitty programmer — starts off pretty good, declines into slop, and you're stuck babysitting instead of doing the architectural work that matters. Structural patterns constrain what it can generate. If your types permit skipping validation, it will skip validation. If your error model permits swallowing failures, it will swallow failures. Not out of malice — out of plausibility.

What allows more deterministic behavior in LLM harnesses also leads to more predictable, stable, safe programs. The constraints are recursive. They govern the program at runtime. They govern the process of writing the program.

What can you do about it? Not better prompts or more instructions — those help, but they don't *constrain*. The LLM can always ignore a prompt. It can't ignore a type error.

Different manifestations, same solution. This series explores both by building something.

The series uses [Effect](https://effect.website) throughout — not as a library preference but as a deliberate structural decision. Effect provides a modular, composable basis for the guardrails we build: schemas that are both types and validators, tagged errors with exhaustive handling, typed effect channels that make side effects visible. Each post shows a pattern in plain TypeScript first, then in Effect — but Effect is the production choice because it composes these patterns without the boilerplate collapsing under its own weight. Choosing Effect is itself a guardrail. It signals to an LLM assisting with development that these patterns are the established mechanism, reducing the surface where ad-hoc solutions can appear in generated code.

## The Darwin Core Archive Agent

Darwin Core is the international standard for biodiversity data — hundreds of terms, specific formatting requirements, relationships between record types like occurrences, events, and measurements. Publishing Darwin Core data means producing a Darwin Core Archive: a zip file containing standardized CSV data, a `meta.xml` descriptor declaring the archive's structure, and an `eml.xml` metadata document in Ecological Metadata Language describing the dataset for cataloguing and discovery.

Scientists end up with validated data — columns mapped to Darwin Core terms, values formatted correctly. The data is ready. What they don't do — or do poorly, inconsistently, and reluctantly — is package it. Writing a standards-compliant EML document means authoring a dataset abstract, describing collection methods, selecting keywords from controlled vocabularies, specifying geographic and temporal coverage, defining citation formats, choosing licenses, and structuring contact information — all according to a complex XML schema that's unforgiving about structure and completeness. It's the metadata equivalent of writing documentation: everyone agrees it matters, almost nobody does it well, and it's exactly the kind of work that gets deferred until a submission deadline forces it.

This problem isn't unique to biodiversity. Any domain with structured publication requirements — research data repositories, regulatory filings, standards-compliant reporting — has the same shape: the hard work is done, the packaging is what people skip.

The Darwin Core Archive Agent is a harness that guides scientists through packaging validated Darwin Core data for publication. It inspects source data files, determines archive structure, infers what metadata it can compute from the data itself, interviews the user for what it can't, generates standards-compliant archive components, validates them, and revises until they're correct.

It's the kind of thing where most implementations are a bag of strings and hope. This one is structurally constrained, where most classes of bug are impossible by construction.

Here's what a single execution looks like, and what can go wrong at every step.

**Source data arrives. The agent inspects already-validated files.** The CSV files contain standardized Darwin Core columns — but the agent can't assume they're well-formed for archiving. Headers might be inconsistent across files, encodings might vary, files that should reference each other might not. This is the first parse boundary: untrusted data crossing into typed territory. The source files are the foundation everything else builds on — if a malformed file gets through, every inference downstream is wrong. *Challenge: how do you validate data at a boundary so that the proof of validation survives into every function that touches it?*

**The agent determines archive structure.** Which file is the core, which are extensions, what's the row type for each. A dataset with occurrences linked to measurements could be structured as an Occurrence core with a MeasurementOrFact extension, or an Event core with both Occurrence and MeasurementOrFact extensions — the choice depends on what the data represents, not just its shape. The LLM reasons about this from the data's content and the user's description. This determination drives every subsequent decision: `meta.xml` structure, which EML sections are required, what cross-validation checks apply. Getting it wrong means an archive that's structurally valid but semantically incorrect — accepted by validators, rejected by aggregators who expect a different core type. *Challenge: how do you parse non-deterministic output into typed, trustworthy structural decisions — or reject them cleanly?*

**The agent gathers metadata it can't infer.** An abstract describing the dataset's purpose. Collection methods. Institutional contact information. License choice. Citation preferences. Keywords from controlled vocabularies. None of this can be computed from the data — it requires the user's domain knowledge and institutional context. The harness must track what's been gathered, what's still missing, and what depends on other answers. The user describes their methods; the harness records it as structured metadata attached to the correct EML section. This isn't a chat — it's a guided interview with typed outputs at every step. *Challenge: how do you make it impossible to finalize metadata that hasn't been gathered — not by policy, but by the absence of a pathway?*

**The agent infers derived metadata from the data.** Geographic bounding box from coordinates. Temporal extent from date columns. Taxonomic coverage from species fields. Record counts per file. These computations are deterministic, but they require reading the source data — and only reading it. During inference, the agent should not write files, modify source data, or interact with external services. The inferred values need confirmation: a geographic bounding box computed from data with outlier coordinates might be wrong, a temporal range might not account for planned future sampling. *Challenge: how do you restrict what the harness can reach — not by policy, but by the absence of a pathway?*

**Confirmation: the agent presents its work for review.** Inferred metadata alongside user-provided metadata, assembled into a complete picture. "I computed a geographic extent of [-48.5, -123.4] to [-47.2, -122.1] from your coordinates — does this represent the study area, or should coverage be broader?" This is a genuine decision point. The computed extent might exclude nearshore areas the study covers. The temporal range might not account for a second season of collection. The user's abstract might contradict the inferred taxonomic scope. These confirmations are structural gates, not rubber stamps — the harness cannot proceed to generation without them. *Challenge: how do you make it impossible to finalize metadata that hasn't been confirmed — not by policy, but by the absence of a pathway?*

**The agent generates the archive.** `meta.xml` declaring file structure, term mappings, and relationships. `eml.xml` containing all metadata in schema-compliant XML. The archive zip itself. During generation, the agent writes to the output directory — and only the output directory. It cannot modify source files. It cannot re-read source data and reinterpret it. The generation phase gets a write capability scoped to the archive output path, nothing else. *Challenge: how do you restrict what the harness can reach — not by policy, but by the absence of a pathway?*

**The agent validates its output.** EML against the EML XML Schema. `meta.xml` against the Darwin Core Archive metafile schema. Cross-validation: do fields declared in `meta.xml` match actual CSV headers? Does taxonomic coverage in EML reflect the species in the data? Are required EML elements present given the archive's content — geographic coverage is required when coordinate data exists, temporal coverage when dates exist? Validation errors are structured — `SchemaViolation` (malformed XML element or missing required attribute), `CrossReferenceViolation` (`meta.xml` declares a column absent from the CSV), `CompletenessViolation` (missing required section given the data's characteristics). Each violation type has a different recovery path. A `SchemaViolation` means the LLM generated invalid XML and needs to fix the structure. A `CrossReferenceViolation` means a mismatch between the descriptor and the data — possibly a misremembered column name. A `CompletenessViolation` might mean the agent needs to go back and ask the user for information it didn't gather. *Challenge: how do you ensure every error variant is handled, and that adding a new one forces every handler to update?*

**The agent revises and re-validates.** EML is complex enough that fixing one section can cascade — restructuring a contact element to satisfy the schema might break the citation that references it. Broadening geographic coverage might require adding new keywords. Fixing a `CrossReferenceViolation` by correcting a column name in `meta.xml` might introduce a `CompletenessViolation` if the corrected column implies additional required metadata. The revision loop must terminate. Stall detection watches for the real failure modes: the same schema violation reappearing after a fix (repetition), no reduction in violation count across rounds (stagnation), the same element toggling between two structures that each satisfy different subsets of the schema (oscillation). *Challenge: how do you make protocol violations impossible and guarantee bounded execution?*

Each of these challenges has a known solution — a pattern that eliminates the problem structurally. This series defines those patterns one at a time, each building a piece of the archive agent's specification. By the final post, the spec is complete, and we assemble the implementation from the pieces the series produced.

This is spec-driven development in practice. Each post defines a component's constraints with enough rigor that the implementation is nearly determined. An LLM generating code against these specs operates in a narrow space where most programs are correct — because the spec eliminates the rest. The series isn't just *about* structural guardrails. It *is* one.

## Why a Custom Harness?

There's a pragmatic objection to all of this: couldn't you just use an LLM with skills? Write a detailed prompt that describes the packaging workflow, provide reference material on the EML schema, and let Claude (or whatever model) walk the user through it conversationally. It would work. For many tasks, it's the right call. Skills are dramatically lower effort than a custom harness, and for workflows where "usually correct" is acceptable, the cost-benefit is clear.

The distinction is between advisory constraints and structural constraints. A skill says "validate EML before finishing." The harness makes it impossible to reach the complete state without passing through validation. A skill says "don't modify source data." The harness doesn't provide a write capability during the inspection phase. The LLM can ignore a skill's instructions — it cannot ignore the absence of a capability in its environment. Skills are comments; the harness is the type system.

The question is when you cross the line from "skill is fine" into "harness is warranted." Several factors push you across.

**Organizational approval and auditability.** A harness can be audited. You can point at the state machine definition and enumerate every possible behavior. You can demonstrate that source data is never modified, that user confirmation is structurally required before generation, that the revision loop terminates. You can't audit a skill the same way — the LLM's compliance is probabilistic. For an institution that needs to demonstrate process rigor for data publication — and research data management increasingly requires this — "we use approved tooling with verified behavior" is a fundamentally different claim than "we prompt the model carefully."

**Provider independence and longevity.** The harness runs against whatever model you wire in. Skills are coupled to the interface they run in and the specific model's ability to follow them. If the provider's behavior changes, if you need to switch models, if you need to run locally or in the field where cloud APIs aren't available — the harness doesn't care. The state machine and capability boundaries are your code, not someone else's feature. An organization building a publication workflow around skills is building on a dependency it doesn't control and can't pin to a version.

**Reproducibility and logging.** Given the same inputs and the same user decisions at confirmation gates, the harness produces the same trajectory through states. The output varies — LLM generation is non-deterministic — but the process is deterministic and fully logged. Every state transition, every parse boundary crossing, every confirmation gate, every revision round with its violation counts and stall signals. A skill-driven conversation produces a chat log. You can read it after the fact, but you can't systematically audit whether the workflow was followed, whether confirmation was obtained before generation, or whether validation actually ran.

**Statefulness across a complex workflow.** The harness knows where it is, what's been established, and what's been tried. It knows which metadata fields were inferred vs. user-provided vs. still missing. If validation fails and requires re-gathering information, it doesn't re-ask what it already knows. Conversation context degrades over long interactions: the model summarizes, compresses, loses track of what was confirmed three exchanges ago. The machine's context is a structured record, not a chat log that gets progressively vaguer.

**Scoped failure and recovery.** When a skill-driven conversation fails — the model generates invalid XML, loses track of a required field, contradicts itself — recovery is "re-read the conversation and figure out what went wrong." When the harness fails, the failure is scoped: a `SchemaViolation` on a specific EML element, a `CrossReferenceViolation` between `meta.xml` and a specific CSV file, a circuit breaker trip with a logged stall signal. The failure tells you what happened and where. The recovery path is determined by the violation type, not by asking the model to introspect on its own mistakes.

None of this means skills are wrong. For a solo researcher packaging a single dataset, a well-written skill might be perfectly adequate. The harness is warranted when the workflow is repeated, institutional, auditable, or high-enough-stakes that "the model usually follows instructions" isn't sufficient. This series builds the harness not because it's the only way — but because it's the way that composes, scales, and provides guarantees.

## The Series

1. **[Types as Receipts](/types-as-receipts)** — Validation that changes the type. Source file inspection produces `ValidatedSource` receipts — proof that files were parsed and their structure confirmed. Metadata gathered from users produces `UserProvided` values distinct from `Inferred` values. Archive structure decisions carry typed proof of how they were determined. Unverified metadata is a type error.

2. **[Making Illegal States Unrepresentable](/making-illegal-states-unrepresentable)** — Discriminated unions and exhaustive matching. `HarnessError` covers every harness-level failure — parse errors, tool failures, budget exhaustion, validation failures. `ArchiveViolation` nests inside it for archive-specific violations: `SchemaViolation`, `CrossReferenceViolation`, `CompletenessViolation`. Each variant carries the data needed for its recovery path, and the compiler enforces handling of every one.

3. **[State Machines and Lifecycle](/state-machines-and-lifecycle)** — The packaging workflow as an XState machine with inspecting, structuring, confirming, gathering, generating, validating, and revising states. The revision loop with domain-specific stall detection — repetition, stagnation, oscillation — replaces flat retry counters. A circuit breaker trips when stall confidence exceeds the threshold. The machine governs what the agent can do at runtime; invalid transitions are absorbed, not crashed.

4. **[Encoding Protocols in State](/encoding-protocols-in-state)** — Typestate patterns for compile-time protocol enforcement. The compiler enforces that metadata gathering and confirmation precede archive generation — skipping the `metadataConfirmed` gate is a type error, not a policy. `initial` -> `dataInspected` -> `structureDetermined` -> `metadataGathered` -> `metadataConfirmed` -> `archiveGenerated` -> `validated`. The `[0,0]` bounding box bug — where an outlier coordinate makes geographic coverage span the entire globe — is structurally impossible because `generateArchive` only accepts `Workflow<"metadataConfirmed">`. Each phase transition is a type-level proof.

5. **[Capabilities and Effects](/capabilities-and-effects)** — Controlling what code can do by passing capabilities explicitly. Inspection gets `FileReader`. Structuring and confirmation get `UserPrompt`. Gathering gets both `UserPrompt` and `FileReader` — user interview plus data-driven inference, read-only. Generation gets `ArchiveWriter` scoped to the output directory. Validation is pure computation — no capabilities required. Each phase's capability set is structurally enforced.

6. **[Specification and Generation](/specification-and-generation)** — The cumulative specification constrains generation. Every piece from Posts 1-5 becomes the mold — we generate the EML metadata composer against it and watch four compiler errors trace to four specific constraints. Three publication profiles (GBIF, OBIS, DataONE) demonstrate scaling. The review surface narrows to three genuine design decisions: abstract quality, keyword selection, and coverage granularity.

7. **[Verification](/verification)** — Property-based testing, model-based testing, and fault injection — each targeting a different architectural layer, each proving the specification holds under adversarial conditions.
